第一个Blind SSRF

第一个Blind SSRF 位于邀请链接页面:

https://www.target.com/give-5-get-5/UOG262

挂上burpsuite作为代理，监控该页面触发的所有流量，发现该页面会发送POST请求给 https://www.target.com/rpc/url-health-check-rpc/check

请求参数为:

url=https%3A%2F%2Fwww.instagram.com%2Fp%2FBuMIheXHKsS%2F&sourceUrl=https%3A%2F%2Fwww.target.com%2Fgive-5-get-5%2FUOG262

返回的响应为:

{"result":null,"output":""}

经过测试，该请求的url 参数存在blind ssrf 漏洞

从请求报文中可以看到，服务端会自动向URL的末尾添加/embed ，导致攻击者无法完全控制URL的path的部分，但是绕过方法很多：

  1. 该特性可以通过30x重定向绕过。
  2. 使用#将追加的/embed 变成锚点部分

这几乎是一个全盲的SSRF漏洞：

  1. 似乎没有任何SSRF filter来检查和过滤用户输入的URL，不需要任何绕过。
  2. 异步请求，无法通过延时来进行内网端口扫描(XSPA)
  3. 没有任何回显
  4. 不支持gopher协议，不支持file协议
  5. 不存在CRLF注入，不能走私请求报文
  6. 会跟随30x重定向

所以单纯利用该SSRF，我们几乎无法知道自己是否成功打到了内网。

但是通过反连请求，我大约收集到5个不同的IP地址，这些IP地址都是AWS的IP地址，所以大致可以推断，目标站点部署在AWS上,并且似乎做了负载均衡

我开始推测为什么访问该页面会触发上述POST请求，于是我审计了这个页面的前端代码，发现该页面使用iframe 嵌入Instagram的一些页面，同时使用js代码来检查这些Instagram页面对应的的URL是否health ，相关js代码如下:

 RPC('url-health-check/check', {url: "https:\/\/www.instagram.com\/p\/BuMIheXHKsS\/", sourceUrl: "https:\/\/www.target.com\/give-5-get-5\/UOG262"});

第二个Blind SSRF(半盲)

我继续检查目标站点的其他功能，发现了第二个SSRF：一个 webhook 模拟器功能，输入一个URL，目标站点会向该URL 发送一个POST请求。但是POST请求的请求体不可控。

这个模拟器功能很有意思，他会向你指定的URL发送POST请求，且SSRF请求的得到的响应报文的响应码会回显到这个api的响应报文中。

由于这个SSRF 所处的位置过于明显，目标站点部署了 SSRF filter 来检查用户输入的URL，经过一番探测，该SSRF filter 似乎采用了黑名单校验机制。比如，我先筛选出该目标的所有403 子域，通过该SSRF漏洞访问，发现所有的403 子域名都在黑名单上。

经过更多的尝试之后，我逐渐掌握了SSRF fitler 的部分黑名单规则:

是否存在 localhost 关键字
http://127.0.0.1
http://192.168.x.x
http://169.254.169.254
https://*.redacted.com
https://*.theredacted.com

不过可以绕过这个SSRF filter的姿势很多:

姿势1：

http://127.1:80

姿势2：将点分十进制表示的IP转化为十进制IP地址绕过

http://2130706433/ = http://127.0.0.1
http://3232235521/ = http://192.168.0.1
http://3232235777/ = http://192.168.1.1
http://2852039166/  = http://169.254.169.254

姿势3： 利用DNS解析

http://customer1.app.my.company.127.0.0.1.nip.io = 127.0.0.1

第三个Blind SSRF

能否结合利用以上两个Blind SSRF呢？比如说:

  1. 利用 webhook SSRF 进行 XSPA，扫描内网中存活的主机和开放的端口
  2. 利用 url-health-check SSRF 发送payload 探测内网中存活的主机是否存在SSRF或者RCE等可以触发反连的漏洞

因为 webhook api 设置了速率限制，大概一小时只能发送100来个请求，所以挑选扫描范围非常重要。我注意到目标站点的部署在AWS上，所以就先使用webhook SSRF 扫描了AWS 默认VPC的内网范围, aws 一个vpc至少需要跨三个区域，所以一个vpc下至少需要有三个子网。
